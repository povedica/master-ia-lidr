Estimador CAG

API de Context-Augmented Generation que transforma transcripciones de reuniones, briefs y adjuntos en estimaciones de software estructuradas. Baseline de AI Engineering con guardrails, caché semántica opcional, sesiones multi-turno y UI React.

FastAPI LiteLLM Instructor Jinja2 CAG React + Vite pytest

1 Visión general

El proyecto implementa un estimador de proyectos de software usando CAG deliberadamente simple: ejemplos few-shot mantenidos por el equipo (no RAG vectorial), inyectados en el system prompt según el modo adaptativo de la petición.

¿Qué hace?

Recibe contexto estructurado (transcripción, metadatos del proyecto, adjuntos) y devuelve una estimación con fases, horas, supuestos y nivel de confianza.

¿Qué NO incluye?

  • Autenticación de producción
  • Persistencia de sesiones entre instancias
  • Moderación de contenido enterprise-grade

Superficies de API

  • v1 — texto Markdown + SSE streaming
  • v2 — JSON estructurado + guardrails
  • sessions — multi-turno con memoria en RAM
  • embeddings/ingest — pipeline Session 07 (presupuestos → vectores en Postgres)
  • search — búsqueda semántica top-k sobre chunks persistidos (feature-038)
  • retrieval — producción A/B/C/D: vector, híbrido RRF, rerank cross-encoder y evaluación (feature-050)
  • retrieval-debug — API interna y pantalla React gated por VITE_ENABLE_RETRIEVAL_DEBUG para explicar ramas vectorial, lexical e híbrida, comparar rankings e inspeccionar chunks (features 042–047)

Clientes

  • Web UI React (web/)
  • curl / Bruno (api-collection/)
  • OpenAPI en /docs

    8 Guardrails

    Pipeline de políticas en v2 y sesiones. Cada guardrail tiene modo de rollout configurable.

    Input — dominio

    Rechaza peticiones fuera de estimación de software. HTTP 200 degraded o skip según política.

    Input — injection

    Heurísticas deterministas sobre el texto del usuario antes de llamar al proveedor.

    Input — PII

    Detección básica de datos personales sensibles en la superficie de assessment.

    Output — confianza

    Umbral mínimo de confidence en EstimationResult.

    Output — leakage

    Heurísticas para detectar filtrado de instrucciones del system prompt.

    ACB Actor-Critic-Boss (orquestación sesión)

    Capa opcional de calidad online solo en POST /api/v1/sessions/{session_id}/estimate. Separa generación (Actor), evaluación estructurada (Critic) y gobernanza del proceso (Boss). Desactivada por defecto (ACB_ENABLED=false).

    No confundir con evals offline. El juez LLM de tests/evals/ mide calidad post-hoc; ACB es orquestación en producción con roles distintos y presupuesto de iteraciones acotado.

    Cómo funciona

    sequenceDiagram
    participant S as SessionService
    participant P as LLMPipeline
    participant O as AcbOrchestrator
    participant A as Actor
    participant C as Critic
    participant B as Boss

    S->>S: merge metadata, messages_override
    alt ACB inactivo
        S->>P: run_structured
    else ACB activo
        S->>P: run_structured_with_acb
        P->>P: input guardrails
        Note over P: acb_cache_bypassed (no cache serve)
        loop hasta ACCEPT o SYNTHESIZE o budget
            P->>O: iteración
            O->>A: estimate_structured (+ revision si aplica)
            A-->>O: EstimationResult
            O->>C: CriticFeedback
            C-->>O: issues estructurados
            O->>B: BossDecision
            B-->>O: accept | revise | synthesize
        end
        P->>P: output guardrails (solo resultado final)
    end
    P-->>S: StructuredPipelineOutcome

    Actor

    Genera el mejor candidato EstimationResult. Reutiliza estimate_structured con prompts ACB; en revisiones inyecta actor_revision.j2 con instrucciones del Boss.

    Critic

    Detecta defectos (completitud, aritmética, coherencia, metadata). Devuelve CriticFeedback — nunca reescribe la estimación.

    Boss

    Decide: accept, revise (una pasada focalizada) o synthesize (presupuesto agotado). La política determinista en policy.py acota el bucle.

    Activación

    Capas: kill switch global, allowlist de endpoint, override por request (orchestration), force en dev local. Escape hatch: single_pass.

    Variables de entorno

    VariableDefaultEfecto
    ACB_ENABLEDfalseActiva ACB en endpoints permitidos
    ACB_ENABLED_ENDPOINTSsession_estimateAllowlist de endpoint lógico
    ACB_MAX_ITERATIONS2Máximo de pasadas Actor (incl. inicial)
    ACB_ALLOW_SYNTHESIZEtrueBoss puede sintetizar al agotar budget
    ACB_BLOCKING_SEVERITIEScritical,majorSeveridades que disparan revisión
    ACB_FORCE_ENABLED_IN_DEVfalseForce ON con APP_ENV=local + DEV_MODE
    DEV_MODEfalseExpone estimate.acb_trace en JSON
    Coste y latencia: un submit con ACB puede implicar 3–9 llamadas LLM según iteraciones y acción del Boss. Usar default-off y activar solo en demos o A/B controlado.

    07 Pipeline de embeddings (Session 07)

    Módulo de aprendizaje aislado bajo app/embedding_pipeline/. Convierte presupuestos históricos JSON en chunks de texto enriquecidos y vectores OpenAI (text-embedding-3-small, 1536 dimensiones). No comparte código con la caché semántica del estimador v2 ni con el flujo CAG.

    Milestone consolidado (features 030–035): schemas → adapter PipelineDocument → chunker markdown → embedder → run_ingest CLI → loader/parser upstream → harness e2e + CLIs ops. Persistencia Postgres (feature-036): servicio postgres (pgvector), modelos documents/chunks, Alembic async. Índice HNSW ix_chunks_embedding_hnsw con vector_cosine_ops (feature-040). Ingest transaccional (feature-037): POST /api/v1/embeddings/ingest persiste un presupuesto por petición vía run_persistent_ingest(); respuesta con metadatos (sin vectores crudos); 409 si source_path duplicado. Búsqueda semántica (feature-038): POST /api/v1/search embebe la query con el mismo modelo, rankea chunks por distancia coseno (<=>) y devuelve top-k. Postgres puede usar índice HNSW ix_chunks_embedding_hnsw (feature-040) sin cambiar el contrato HTTP. Retrieval debug (features 042–048): POST /api/v1/retrieval-debug devuelve ramas vectorial, lexical, híbrida y rerank placeholder: la vectorial rankea por distancia coseno; la lexical usa websearch_to_tsquery sobre chunks.content_tsv + ts_rank_cd, con índice GIN ix_chunks_content_tsv_gin, pg_trgm/ix_chunks_content_trgm y matched_terms; filters acota candidatos por document_type, sector, tecnología, fuente, idioma, tags y año antes de rankear; la híbrida fusiona con RRF o weighted fusion; y rerank.enabled=true ejecuta NoOpReranker por defecto, conserva el orden, llena branches.rerank/rerank_rank y avisa que es un placeholder. Los filtros escalares usan JSONB @> sobre chunks.metadata y reutilizan ix_chunks_metadata_gin; tags exige contains-all y year usa rango inclusivo. GET /api/v1/retrieval-debug/chunks/{id} muestra contenido completo, vecinos y documento padre. Comparación vs estimator: docs/work-items/adr-001-embedding-pipeline-vs-estimator-ingestion.md.

    Vista funcional

    ¿Qué problema resuelve?

    Preparar texto de presupuestos de software (componentes, stack, sector, horas) en fragmentos embeddables listos para búsqueda semántica o clustering — sin acoplar aún a Redis ni a un vector store.

    Dos formas de uso

    • Ingest HTTP persistido: POST /api/v1/embeddings/ingest — un presupuesto por petición → filas en documents/chunks + metadatos de ingestión (sin vectores en la respuesta).
    • Búsqueda semántica: POST /api/v1/search — query en lenguaje natural → top-k chunks rankeados por distancia coseno.
    • Debug retrieval interno: POST /api/v1/retrieval-debug y GET /api/v1/retrieval-debug/chunks/{id} — observabilidad de ramas vectorial/lexical/híbrida, filtros de metadata, términos encontrados, diff de rankings y contexto completo del chunk.
    • Ingest desde directorio: python -m app.scripts.ingest_from_dir --dir … — JSON en disco → run_ingest en memoria (o --dry-run).
    • Compare ad hoc: python -m app.scripts.compare — dos textos → similitud coseno.

    Entrada típica

    JSON de presupuesto con budget_id, metadatos de cliente, resumen de proyecto y lista de components (AUTH, API, DB, etc.).

    Salida típica

    HTTP (feature-037): PersistentIngestResponse con document_id, chunks_created, embedding_dimension, ingestion_time_ms. Los vectores quedan en Postgres (chunks.embedding). CLI run_ingest sigue devolviendo EmbeddedChunk en memoria.

    10 Sesiones multi-turno

    Estado en memoria (app/services/sessions.py). No apto para multi-instancia sin almacenamiento externo.

    Memoria conversacional

    Ventana deslizante de pares user/assistant pasados al LLM estructurado via messages_override. El prompt actual incluye transcript + adjuntos completos.

    Metadata derivada

    Heurísticas extraen nombre, tipo, audiencia del formulario + transcript + texto de adjuntos. Se fusiona con el snapshot anterior de la sesión.

    Limitación de producción: las sesiones se pierden al reiniciar el proceso y no se comparten entre workers Uvicorn. Para producción real se necesitaría Redis/PostgreSQL + sticky sessions o IDs externos.
    Verificación: test_sessions_integration.py valida memoria, metadata y adjuntos (mecánica). La calidad de la estimación multi-turno se mide en tests/evals/ — ver Evals sesión (§15).

    11 Consumo de tokens y coste

    Estimación orientativa para una petición típica con gpt-4o-mini. Valores reales dependen del tamaño del transcript, adjuntos y modo.

    System prompt (instrucciones + few-shot)~1.800 – 4.500 tokens
    User prompt (transcript + metadatos + adjuntos)~500 – 8.000+ tokens
    Output estructurado (Instructor JSON)~800 – 2.048 tokens
    Preprocessing two_phase (si activo)~300 – 1.500 tokens extra
    Embedding caché semántica (si activa)~1 llamada embedding / request
    Ahorro en cache hit~95–100% tokens LLM evitados

    12 Partes críticas del sistema

    Módulos cuyo fallo impacta directamente la funcionalidad core o la seguridad.

    EstimationService + LLMPipeline

    Orquestadores centrales. Un bug aquí afecta todas las superficies (v1, v2, sesiones).

    structured_llm_client.py

    Puerta de entrada a Instructor. Validación de roles en messages y reintentos ante schema mismatch.

    llm_chain.py

    Cadena de fallback. Configuración incorrecta de keys → 503 o degraded mode.

    estimation_engine.py

    Routing determinista de modo. Afecta profundidad y validación de la estimación.

    prompt_renderer + Jinja2 templates

    Cualquier error de template rompe el render en runtime (StrictUndefined).

    sessions.py (in-memory store)

    Pérdida de estado en restart. Race conditions potenciales con múltiples workers.
